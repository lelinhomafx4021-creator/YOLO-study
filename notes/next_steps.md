## 📅 阶段一：吃透 Baseline 模型指标，学会“诊断”模型（当前核心）

* **目标**：不当“只管训练不管效果”的调参侠，要学会看懂你的模型到底在哪些类别上表现好，哪些表现差。
* **具体要做的事**：
  1. 运行你的 [03_val.py](file:///d:/vision_algo_workspace/vision-bootcamp/01_helmet_detect/03_val.py)，仔细观察终端打印出的 `helmet`（戴帽）、`no-helmet`（未戴帽）、`safety-vest`（安全背心）这三个类别的单独 `Precision`（精准率）和 `Recall`（召回率）。
  2. 找出当前模型的弱点：到底是哪一个类别的指标比较低？是漏检偏多（Recall低），还是误检偏多（Precision低）？
  3. 打开 `runs/safety_helmet/yolo11n_baseline_v1/` 目录，找到并点开 `confusion_matrix_normalized.png`（归一化混淆矩阵图），看看模型最容易把什么类别认成什么类别。
* **🎓 面试考点剧透**：
  * **面试官问**：“如果你的模型在工地现场经常把背景里的一个圆球或者路障误检成安全帽，你会怎么解决？”
  * **大白话回答**：“这叫**误检（False Positive）**。第一，我可以去数据集里看看是不是背景里没有包含这些干扰物，如果数据集太单一，我就需要补充一些‘负样本’（即只有背景、路障但没有安全帽的图片）放到训练集里，告诉模型这些不是安全帽；第二，我可以在推理（predict）时提高置信度阈值（比如 `conf=0.5`），过滤掉低置信度的误检；第三，可以用数据增强（如 Mosaic）来提升小目标的分类能力。”

## 📅 阶段二：把“坏样例（Bad Cases）”可视化出来（直观调优）

* **目标**：光看数字指标很空洞，我们必须亲眼看到模型在哪些图片上“翻车”了，这在工业界叫 **Bad Case 分析**。
* **具体要做的事**：
  1. 挑出几张你觉得背景复杂、或者光线很不好的测试图片，放到一个测试目录。
  2. 用我们之前写的定制画框脚本 [05_pro_render.py](file:///d:/vision_algo_workspace/vision-bootcamp/01_helmet_detect/05_pro_render.py)，把推理出来的图保存下来。
  3. 亲眼去数一数：模型有没有漏掉远处的戴帽工人？有没有把红色的衣服错认成安全帽？
  4. 总结出你的 Baseline 模型的“三大硬伤”（比如：逆光看不清、小目标容易漏、红衣服容易认错）。
* **🎓 面试考点剧透**：
  * **面试官问**：“你做完项目后，是怎么对模型进行针对性优化的？”
  * **大白话回答**：“我没有盲目去改网络结构。我先把验证集里模型识别错的图片（Bad Cases）全部导出来分析，发现模型在**远距离小目标**和**工人背对摄像头**时漏检严重。于是我针对性地调整了训练时的输入尺寸（从 640 提高到 800 试试），并且开启了 Mosaic 数据增强（把 4 张图拼成 1 张，人为增加小目标比例），最终把未戴帽类别的 Recall 提升了约 4%。”

## 📅 阶段三：模型轻量化部署（ONNX 导出与极速推理）

* **目标**：脱离笨重的 PyTorch 框架，用工业界最常用的 **ONNX 运行时** 跑模型，这是算法落地部署的必备技能。
* **具体要做的事**：
  1. 用一行代码把我们的 PyTorch 模型（`.pt`）导出为通用格式（`.onnx`）。
  2. 编写一个不需要安装 PyTorch、只安装了 `onnxruntime` 和 `opencv` 的轻量级推理脚本 `onnx_inference.py`。
  3. 手写图像的预处理逻辑（Letterbox 缩放、归一化、通道转换）和后处理逻辑（NMS 非极大值抑制的调用），用 ONNX 跑通图片检测。
* **🎓 面试考点剧透**：
  * **面试官问**：“为什么模型部署要导出成 ONNX？你在导出和推理时遇到过什么坑？”
  * **大白话回答**：“第一，ONNX 是个通用格式，方便我们用 C++ 或者其他更快的推理引擎（如 TensorRT、ONNX Runtime）加载，摆脱了对 PyTorch 的依赖，包体积小、速度快；第二，部署最容易踩的坑是**精度对齐**。如果我们在 Python 中用 OpenCV 做预处理时（比如把图像缩放到 640x640 并做归一化）的细节，和训练时 PyTorch 内部做的预处理对不上（比如忘记把 BGR 转成 RGB，或者除以 255 没做好），ONNX 推理出来的框就会满屏乱飞。我当时通过用 `numpy` 严格对齐两边的输入矩阵数值，解决了这个问题。”
