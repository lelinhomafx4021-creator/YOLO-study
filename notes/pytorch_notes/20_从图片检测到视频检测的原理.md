# 20_从图片检测到视频检测的原理

---

## 💡 一、 核心结论（一句话讲透）

**视频检测 = 图片检测 + 一个 while 循环。**

视频的本质就是一堆按时间顺序排列的图片（帧）。所谓"视频检测"，就是用 OpenCV 不断地从视频里**抽出一帧图片**，把它扔给我们已经写好的图片检测代码跑一遍推理，把画好框的图片再**塞回去显示到屏幕上**。如此循环往复，人眼看到的就是一段"实时检测"的视频了。

---

## 🎬 二、 视频到底是什么？（物理本质）

* 一段 30fps 的视频，意味着**每秒钟包含 30 张图片**。
* 这些图片一张接一张快速播放，因为人眼有"视觉暂留"效应（大约 1/24 秒），所以看起来就像是连续的运动画面。
* 所以，**摄像头拍出来的"视频流"，在计算机内部就是一条源源不断的"图片流水线"**。

> **大白话比喻**：
> 你可以把视频想象成一本**翻页动画书**。每一页就是一帧图片。我们要做的事情就是：翻一页 ➡️ 对这页图进行安全帽检测 ➡️ 把检测结果画在这页上 ➡️ 展示给用户看 ➡️ 翻下一页 ➡️ 重复。

---

## 🔁 三、 视频检测的代码骨架（伪代码）

```text
1. 打开视频源（可以是本地视频文件，也可以是摄像头）
2. 加载模型（加载我们的 best.onnx 或 best.pt）

3. while True:  # 无限循环
4.     从视频源中"抓取"一帧图片 (frame)
5.     如果视频播完了（没有帧可取了）：
6.         break 退出循环
7.
8.     把这一帧图片送进模型跑一次推理（和我们之前做图片检测一模一样）
9.     在这一帧图片上画出检测框
10.
11.    【业务逻辑】如果检测到 no-helmet：
12.        在画面上叠加红色警告文字
13.        把当前这帧截图保存到 alarm_logs/ 文件夹（取证）
14.
15.    把画好框的图片显示到屏幕窗口上（cv2.imshow）
16.    等待用户按键（按 q 退出）
17.
18. 释放视频资源，关闭窗口
```

## 📐 四、 对应的真实 Python 代码骨架

```python
import cv2
from ultralytics import YOLO

# 1. 加载模型（可以是 .pt 也可以是 .onnx，YOLO 自动识别）
model = YOLO("best.onnx")

# 2. 打开视频源
#    如果是本地视频文件：cv2.VideoCapture("工地监控.mp4")
#    如果是电脑自带摄像头：cv2.VideoCapture(0)  ← 0 代表第一个摄像头
cap = cv2.VideoCapture("工地监控.mp4")

# 3. 逐帧循环
while cap.isOpened():
    ret, frame = cap.read()  # ret=是否成功读到帧, frame=这帧图片的矩阵
    if not ret:
        break  # 视频读完了，退出

    # 4. 把这一帧扔给模型跑推理（和图片检测完全一样）
    results = model.predict(source=frame, conf=0.25, save=False)

    # 5. 在这一帧上画框（YOLO 提供了内置的 plot 方法）
    annotated_frame = results[0].plot()

    # 6. 把画好框的帧显示到屏幕窗口上
    cv2.imshow("SafetyHelmet Detection", annotated_frame)

    # 7. 等待 1 毫秒，如果用户按了 'q' 键就退出
    if cv2.waitKey(1) & 0xFF == ord('q'):
        break

# 8. 释放资源
cap.release()
cv2.destroyAllWindows()
```

## 🧠 五、 关键 API 解释

### `cv2.VideoCapture(source)`
* **作用**：创建一个"视频读取器"对象。
* **参数**：
  * `"xxx.mp4"`：读取本地视频文件。
  * `0`：打开电脑自带的第一个摄像头（笔记本前置摄像头）。
  * `"rtsp://..."` 或 `"http://..."` ：连接网络摄像头的实时视频流（工地监控就是这种）。

### `cap.read()`
* **作用**：从视频中**抽取下一帧图片**。
* **返回值**：
  * `ret`（布尔值）：是否成功读到了一帧。如果视频播放完毕或摄像头断开，`ret` 会变成 `False`。
  * `frame`（numpy 数组）：这一帧图片的像素矩阵，格式和 `cv2.imread()` 读出来的完全一样，都是 `(H, W, 3)` 的 BGR 矩阵。

### `cv2.imshow(window_name, image)`
* **作用**：弹出一个桌面窗口，把传入的图片实时显示在窗口里。
* **注意**：必须配合 `cv2.waitKey()` 使用，否则窗口会一闪而过看不到。

### `cv2.waitKey(delay_ms)`
* **作用**：程序暂停 `delay_ms` 毫秒，等待用户按键。
* **为什么传 `1`**：因为我们希望每帧只暂停 1 毫秒就立刻处理下一帧，保持视频的流畅播放。如果传 `0`，程序会一直卡在这里等你按键（变成了一帧一帧的手动翻页模式）。

## ⚡ 七、 工业界的性能瓶颈与优化（进阶知识）

在实际工地的 100 路摄像头场景下，"逐帧推理"会遇到一个严重问题：**显卡跑不过来。**

比如 30fps 的视频流，意味着每秒要推理 30 次。如果模型推理一次需要 50 毫秒（即每秒最多跑 20 次），那视频就会越来越卡，最终延迟达到好几秒。

#### 工业界常见的优化思路：
1. **跳帧检测**：不是每帧都推理，而是每隔 3~5 帧才推理一次。中间跳过的帧直接显示上一次的检测结果。
2. **多线程/异步**：一个线程负责不停地从摄像头读帧（I/O 线程），另一个线程负责跑模型推理（计算线程）。两者并行执行，互不阻塞。
3. **模型加速**：将 ONNX 模型进一步转换为 TensorRT (`.engine`) 格式，利用显卡的硬件级加速把单帧推理时间从 50ms 压到 5ms。
