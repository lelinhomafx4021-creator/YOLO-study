# 视频推理实战：跳帧、EMA 平滑与异步线程

## 这一节在学习顺序里的位置

前面你已经知道：

- 视频就是连续的图片，一帧一帧读
- `cv2.VideoCapture` 读摄像头/视频
- `model.predict()` 对每帧做推理

但有一个根本问题：

**推理比播放慢怎么办？**

摄像头 30 帧/秒（每帧 33ms），YOLO 推理一帧 50ms。每帧都推理 → 来不及。

这一节讲三种解决办法，从简单到复杂。

---

## 先说最核心的一句话

**视频推理的流畅感来自两件事：少算一点，或者别让显示等推理。**

---

## 1. 为什么推理比播放慢

```
摄像头: 30 帧/秒 → 每帧间隔 33ms
YOLO:   推理一帧 → 50ms

每帧都推理:
  帧1 推理(50ms) → 显示 → 帧2 推理(50ms) → 显示
  每帧 52ms → 每秒只能处理 19 帧
  摄像头送来了 30 帧 → 11 帧被丢掉
```

**推理是瓶颈。摄像头送得快，GPU 算不过来。**

---

## 2. 办法一：跳帧

最简单。每 N 帧推理一次，中间帧不推理。

```python
skip = 3
if frame_count % skip == 1:
    result = model.predict(frame)   # 推理帧
    cache = result.plot()
else:
    result = cache                  # 跳帧：用缓存
```

```
帧1: 推理 → 框在 (100, 50)
帧2: 跳过 → 框还在 (100, 50) → 人已经走了
帧3: 跳过 → 框还在 (100, 50)
帧4: 推理 → 框跳到新位置
```

3 帧推理 1 次 → 推理量降到 1/3 → 理论快 3 倍。

**但框会"粘"在旧位置。** 跳帧越多，粘得越久。

---

## 3. 跳帧的缓存方式

关卡 5 的代码里，缓存的是**整张画好框的旧图**：

```python
last_annotated = result.plot()  # 画好框的图片
# ...
right = last_annotated           # 跳帧时显示旧图
```

**不是"在当前帧上画旧框"，是直接显示上一帧的旧图。** 框和画面都冻住了。

30fps 下帧间变化小，肉眼很难察觉。但快速移动或 skip 设大（10 帧），就会明显定格跳。

---

## 4. 办法二：EMA 平滑 FPS 显示

FPS 数字每帧都在变。不用 EMA 的话：

```
屏幕上: 22 → 46 → 12 → 37 → 5 → 51   ← 疯狂跳，没法看
```

EMA 的做法：新值只占 10%，旧值占 90%。

```python
fps = 0.9 * fps + 0.1 * current_fps
```

```
屏幕上: 22 → 24 → 25 → 23 → 22 → 24   ← 稳定，知道大概 20 多
```

**EMA 只影响显示，不影响推理，不影响检测。** 就是让 FPS 数字别跳太狠。

### 为什么叫"指数移动平均"

旧值每帧被乘 0.9。10 帧前的值被乘了 0.9 的 10 次方 = 0.35，还有影响。50 帧前被乘了 0.9 的 50 次方 ≈ 0.005，几乎没了。

**越旧的帧影响越小，像指数衰减一样。所以叫"指数"移动平均。**

---

## 5. 办法三：异步推理

跳帧的问题：推理帧还是要等 50ms，显示就卡一下。

异步的思路：**把推理扔到另一个线程，主线程只管显示。**

```python
主线程（快）:                       推理线程（慢）:
  while True:                       while True:
    frame = cap.read()               拿最新帧
    显示上一轮的推理结果              model.predict(frame)
    cv2.waitKey(1)                   画框 → 把结果传回来
```

**主线程永远只做 `cap.read()` + `imshow()` → 永远流畅。推理线程慢慢跑，跑完更新结果。**

推理再慢也不会卡画面，只是框会滞后几帧。

---

## 6. 异步的三个关键工具

### 6.1 `threading.Thread` — 开子线程

```python
import threading

t = threading.Thread(target=worker_func, daemon=True)
t.start()
# worker_func 在另一个线程里同时跑
# daemon=True: 主线程退出时自动杀掉子线程
```

### 6.2 `Lock` — 防止两个线程同时改同一个变量

```python
lock = threading.Lock()

with lock:
    data = shared_var   # 加锁读写 → 安全
```

两个线程同时改一个变量 → 内容可能是坏的。加锁 = 同一时刻只允许一个线程读写。

### 6.3 `nonlocal` — 闭包内赋值外层变量

```python
def main():
    result = None

    def worker():
        nonlocal result    # 声明：我要改外层的 result
        result = "新值"    # 改变了外层
```

**读不需要 nonlocal，赋值才需要。** 不加的话 Python 当成新建局部变量。

---

## 7. 三种办法的对比

```
跳帧:     推理量少 → 省算力 → 框会粘
EMA:      让 FPS 数字不跳 → 纯显示优化
异步:     推理和显示分开 → 显示永远流畅 → 框滞后但不卡
```

**你的 RTX 3050 跑 yolo11n: 推理 ~15ms/帧，摄像头 33ms/帧 → GPU 比摄像头快 → 逐帧推理就实时了 → 不需要跳帧也不需要异步。**

这些策略是给 CPU 推理、嵌入式设备、多路摄像头准备的。

---

## 8. 这一节最该记住的 5 句话

1. **推理是视频检测的瓶颈，摄像头送得快、GPU 算不过来**
2. **跳帧 = 少推理，框会粘在旧位置**
3. **EMA = 让 FPS 数字不跳，不影响真实性能**
4. **异步推理 = 推理和显示分两个线程，显示永远流畅**
5. **GPU 够强就不需要这些，逐帧推理就是实时**

## 9. 学完这一节后你应该能回答的问题

1. 为什么推理比播放慢
   - 摄像头 30 FPS = 每帧只有 33ms 可用。YOLO 在 RTX 3050 上的 PyTorch 推理 ~50ms/帧 → 跟不上。所以视频看起来"掉帧"或卡顿。本质: 显示等推理，推理等 GPU，显示被拖慢。

2. 跳帧的框为什么会"粘"
   - 跳过帧不跑检测 → 框的位置保持上一帧检测结果不变。人物在走但框停着 → 看起来"粘住了"。用卡尔曼预测可以让框在跳过帧上跟着物体运动（通过速度外推），减轻粘滞感。

3. EMA 的系数 0.9/0.1 代表什么
   - fps = 0.9×old_fps + 0.1×new_fps。0.9 = 旧值权重（惯性大，变化慢），0.1 = 新值权重（反应慢但稳）。高惯性 → 数字不跳，看着舒服；低惯性 → 反应快但抖动。很像汽车避震 — 越硬越精准但颠簸。

4. 异步推理用什么 Python 工具实现
   - threading.Thread: 推理线程独立运行，主线程只管显示。推理线程和主线程通过共享变量通信(最新帧 + 锁)。import threading; Thread(target=推理函数, args=(...)).start()。

5. Lock 是干什么的，不加会怎样
   - Lock（锁）= 只允许一个线程同时修改共享变量。推理线程写 `latest_result` 时，显示线程同时读 → 数据可能错乱（半个 old+半个 new）。加锁: 写前 lock acquire，写后 release → 显示线程等写完了才能读 → 数据安全。

6. nonlocal 是干什么的，不加会怎样
   - Python 里嵌套函数想修改外层函数的变量，需要用 nonlocal 声明。不加 → Python 认为你在内层创建了一个同名局部变量，外层变量不变。加 nonlocal → 内层修改的就是外层那个变量。

7. 什么情况下不需要跳帧和异步
   - GPU 够强（如 RTX 4090 + TensorRT FP16）→ 每帧推理 < 30ms → 逐帧检测就是实时。或者推理目标简单（分类模型 5ms/帧）→ 本就追得上摄像头帧率。跳帧和异步是在"硬件不够"时的优化手段。

---

## 复习速答

- `推理瓶颈`：摄像头帧率和模型推理速度不匹配。
- `跳帧`：降低推理压力，但框会有滞后。
- `EMA`：平滑 FPS 或检测结果，减少抖动。
- `异步线程`：读帧和推理分开跑。
- `Lock/nonlocal`：线程和闭包里的同步工具。
